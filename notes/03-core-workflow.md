## 3a. The Core Workflow

```
┌─────────┐     ┌──────────┐     ┌──────────┐     ┌─────────┐     ┌──────────┐
│  Write  │ ──> │   Init   │ ──> │   Plan   │ ──> │  Apply  │ ──> │ Destroy │
│ (Code)  │     │ (Setup)  │     │ (Review) │     │ (Deploy)│     │ (Teardown)│
└─────────┘     └──────────┘     └──────────┘     └─────────┘     └──────────┘
```

1. **Write** — Author infrastructure as code in `.tf` files
2. **Initialize** — `terraform init` downloads providers and modules
3. **Plan** — `terraform plan` previews changes before applying
4. **Apply** — `terraform apply` executes the planned changes
5. **Destroy** — `terraform destroy` removes all managed resources

### Workflow Best Practices

```bash
# Recommended daily workflow
terraform fmt          # Format code
terraform validate     # Validate syntax
terraform plan         # Review changes
terraform apply        # Apply changes

# CI/CD pipeline
terraform fmt -check   # Fail if not formatted
terraform validate     # Check syntax
terraform plan         # Generate plan
terraform apply        # Apply (with approval gate)
```

## 3b. Initialize a Terraform Working Directory (`terraform init`)

### What `terraform init` Does

1. Downloads provider plugins specified in `required_providers`
2. Downloads module sources
3. Initializes the backend for state storage
4. Creates the `.terraform/` directory
5. Creates or updates `.terraform.lock.hcl`

### Common Usage

```bash
# Basic initialization
terraform init

# Force upgrade of providers beyond lock file
terraform init -upgrade

# Initialize with backend configuration from a file
terraform init -backend-config=backend.hcl

# Migrate state to a new backend
terraform init -migrate-state

# Reconfigure backend (do NOT migrate state)
terraform init -reconfigure

# Use a local plugin directory
terraform init -plugin-dir=/usr/local/share/terraform/plugins

# Disable color output
terraform init -no-color

# Show backend migration plan without executing
terraform init -migrate-state -backend-config=backend.hcl
```

### When to Run `terraform init`

- After cloning a repository
- After adding or changing a `required_providers` block
- After adding or changing a module `source`
- After changing the backend configuration
- After modifying the `terraform` block

### The `.terraform/` Directory

```
.terraform/
├── providers/           # Downloaded provider plugins
│   └── registry.terraform.io/
│       └── hashicorp/
│           ├── aws/
│           └── azurerm/
└── modules/             # Downloaded modules (if any)
```

**Important:** Do NOT commit `.terraform/` to version control. It is regenerated by `terraform init`.

```bash
# .gitignore
.terraform/
*.tfstate
*.tfstate.*
```

**However,** DO commit `.terraform.lock.hcl` to ensure consistent provider versions.

## 3c. Validate Configuration (`terraform validate`)

### What `terraform validate` Does

- Checks syntax of `.tf` files
- Verifies internal consistency of configuration
- Checks that required arguments are present
- Validates variable types and references

**Important:** `terraform validate` runs WITHOUT accessing remote APIs. It only checks the configuration locally.

```bash
# Basic validation
terraform validate

# JSON output
terraform validate -json
```

### Validation Example

```hcl
# This will FAIL validation — missing required argument
resource "aws_instance" "web" {
  # ami is missing!
  instance_type = "t2.micro"
}

# Error: Missing required argument
# "ami" is required but no definition was found.
```

```hcl
# This will PASS validation
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"
}
```

### `validate` vs Other Checks

| Check | Command | What It Checks | Remote Access |
|-------|---------|----------------|---------------|
| Syntax | `terraform validate` | HCL syntax, references, types | No |
| Format | `terraform fmt` | Code style and formatting | No |
| Feasibility | `terraform plan` | Actual changes against real infrastructure | Yes |

**Best practice:** Run `terraform fmt` and `terraform validate` before every commit.

## 3d. Generate and Review an Execution Plan (`terraform plan`)

### What `terraform plan` Does

1. Reads the current state
2. Compares it with the desired configuration
3. Determines what actions are needed
4. Presents an execution plan

```bash
# Basic plan — preview changes
terraform plan

# Save plan to a file
terraform plan -out=tfplan

# Plan with variables
terraform plan -var="instance_type=t2.large"
terraform plan -var-file="production.tfvars"

# Plan for destruction
terraform plan -destroy

# Detailed exit codes
terraform plan -detailed-exitcode
# Exit codes:
#   0 = Success, no changes (empty diff)
#   1 = Error
#   2 = Success, changes present (non-empty diff)

# Refresh-only mode (update state without making changes)
terraform plan -refresh-only

# Skip refreshing state
terraform plan -refresh=false

# Target specific resources
terraform plan -target=aws_instance.web -target=aws_security_group.web

# Plan with replaced resources
terraform plan -replace=aws_instance.web
```

### Plan Output Symbols

```
+ resource "aws_instance" "web" {   # Will be created
  - resource "aws_instance" "old" {  # Will be deleted
  ~ resource "aws_instance" "web" {  # Will be updated in-place
-/+ resource "aws_instance" "web" {  # Will be destroyed and recreated
```

| Symbol | Meaning |
|--------|---------|
| `+` | Resource will be created |
| `-` | Resource will be deleted |
| `~` | Resource will be updated in-place |
| `-/+` | Resource will be destroyed and recreated (forced replacement) |

### Dependency Graph

Terraform builds a dependency graph to determine the order of operations:

```bash
# Generate a visual dependency graph
terraform graph | dot -Tpng > graph.png

# Types of graphs
terraform graph                    # Plan graph (default)
terraform graph -type=plan         # Plan graph
terraform graph -type=plan-destroy # Destroy plan graph
```

**How Terraform resolves dependencies:**
1. **Implicit dependencies** — Terraform detects when one resource references another
2. **Explicit dependencies** — `depends_on` forces dependency order
3. **Parallel execution** — Resources without dependencies are created in parallel

```hcl
# Implicit dependency — Terraform knows VPC must exist first
resource "aws_subnet" "main" {
  vpc_id = aws_vpc.main.id  # Implicit dependency on aws_vpc.main
}

# Explicit dependency — when Terraform can't detect it automatically
resource "aws_instance" "web" {
  depends_on = [aws_iam_role_policy.example]
}
```

### Applying a Saved Plan

```bash
# Save the plan
terraform plan -out=tfplan

# Apply the saved plan (no re-planning needed)
terraform apply tfplan
```

**Note:** A saved plan is a complete snapshot. It includes the configuration, state, and variables at the time the plan was created. This ensures the exact changes reviewed are the changes applied.

## 3e. Apply Changes (`terraform apply`)

### What `terraform apply` Does

1. Creates an execution plan (if not applying a saved plan)
2. Asks for confirmation (unless `-auto-approve`)
3. Executes the plan to reach the desired state
4. Updates the state file with the results

```bash
# Basic apply (shows plan, asks for confirmation)
terraform apply

# Apply without confirmation (use carefully!)
terraform apply -auto-approve

# Apply a saved plan
terraform apply tfplan

# Apply with variables
terraform apply -var="instance_type=t2.large"
terraform apply -var-file="production.tfvars"

# Apply and replace a specific resource
terraform apply -replace=aws_instance.web

# Target specific resources
terraform apply -target=aws_instance.web

# Apply with limited parallelism
terraform apply -parallelism=5

# Refresh-only mode (update state, no infrastructure changes)
terraform apply -refresh-only
```

### `-replace` (Replaces Deprecated `terraform taint`)

```bash
# OLD WAY (deprecated — do not use)
terraform taint aws_instance.web
terraform apply

# NEW WAY (Terraform 0.15.2+)
terraform apply -replace=aws_instance.web
```

**Why `taint` was deprecated:** `taint` modified state to mark resources for replacement, which caused confusion and was error-prone. The `-replace` flag is explicit and declarative.

### Using `-auto-approve`

```bash
# Only use -auto-approve in automation or when you're certain
terraform apply -auto-approve

# In CI/CD — apply a pre-reviewed saved plan
terraform apply tfplan
```

**Warning:** Only use `-auto-approve` in CI/CD pipelines where the plan has already been reviewed, or when you are certain about the changes.

## 3f. Destroy Infrastructure (`terraform destroy`)

### What `terraform destroy` Does

1. Creates a destroy plan for all resources
2. Asks for confirmation
3. Destroys all managed resources
4. Updates the state file

```bash
# Basic destroy (asks for confirmation)
terraform destroy

# Destroy without confirmation
terraform destroy -auto-approve

# Destroy specific resource
terraform destroy -target=aws_instance.web

# Create a destroy plan for later review
terraform plan -destroy -out=destroy.tfplan
terraform apply destroy.tfplan

# Destroy with variable specification
terraform destroy -var="force_destroy=true"
```

**Important:**
- `terraform destroy` is **irreversible**
- It destroys ALL resources managed by the configuration
- Use `-target` to destroy specific resources
- Always review the destroy plan carefully
- State is updated after destruction

### Targeted Operations

```bash
# Apply only specific resources
terraform apply -target=aws_instance.web -target=aws_security_group.web

# Destroy only specific resources
terraform destroy -target=aws_instance.web
```

**Warning:** Targeted operations should be used sparingly. They can cause state drift because dependent resources may not be included.

## 3g. Apply Formatting and Style Adjustments (`terraform fmt`)

### What `terraform fmt` Does

- Rewrites `.tf` files to a canonical format
- canonical indentation (2 spaces)
- Consistent quoting style
- Proper spacing around operators
- Aligns equals signs in blocks

```bash
# Format all .tf files in current directory
terraform fmt

# Format recursively in all subdirectories
terraform fmt -recursive

# Check if files are formatted (CI/CD — returns 1 if not formatted)
terraform fmt -check

# Show diffs (what would change)
terraform fmt -diff

# Format a specific file
terraform fmt main.tf
```

### Before and After `terraform fmt`

**Before:**
```hcl
resource "aws_instance" "web" {
ami="ami-0c55b159cbfafe1f0"
instance_type     =    "t2.micro"
tags={
Name = "WebServer"
}
}
```

**After:**
```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  tags = {
    Name = "WebServer"
  }
}
```

### CI/CD Integration

```yaml
- name: Terraform Format
  run: terraform fmt -check -recursive
```

**Best practice:** Run `terraform fmt` before every commit, and use `terraform fmt -check` in CI pipelines.

## Exam Tips

1. **Workflow order** — Write → Init → Plan → Apply (always init first!)
2. **`plan -detailed-exitcode`** — 0=no changes, 1=error, 2=changes present
3. **`-replace` replaces `taint`** — `terraform taint` is deprecated
4. **`-refresh-only` replaces `refresh`** — `terraform refresh` is superseded by `terraform apply -refresh-only`
5. **`validate` is local** — does NOT access remote APIs
6. **`fmt -check`** in CI to ensure formatting
7. **Saved plans are snapshots** — they include config, state, and variables at plan time
8. **`-target` is for emergencies** — not for regular workflow
9. **Dependency graph** — `terraform graph` visualizes resource dependencies

## Quick Review Questions

1. **Q:** What is the correct order of the core Terraform workflow?
   **A:** Write → Init → Plan → Apply

2. **Q:** What does `terraform init -upgrade` do?
   **A:** Upgrades providers and modules to the newest versions matching constraints, ignoring the lock file.

3. **Q:** What is the difference between `terraform validate` and `terraform plan`?
   **A:** `validate` checks syntax and internal consistency locally; `plan` compares configuration against real infrastructure and requires provider access.

4. **Q:** What replaces `terraform taint`?
   **A:** `terraform apply -replace=<resource_address>`

5. **Q:** What does `terraform plan -detailed-exitcode` return when there are changes?
   **A:** Exit code 2 (exit code 0 = no changes, exit code 1 = error).

6. **Q:** Why should you commit `.terraform.lock.hcl`?
   **A:** To ensure all team members and CI use the same provider versions.

7. **Q:** What does `terraform fmt -check` do?
   **A:** Returns a non-zero exit code if files are not properly formatted; used in CI pipelines.
